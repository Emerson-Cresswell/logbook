================================================================================
1. PROJECT SUMMARY
================================================================================
MyLogbook is a medical procedure and CPD logbook Progressive Web App. It is currently a static HTML/CSS/JavaScript PWA using localStorage, a manifest, and a service worker. It is currently hosted on GitHub Pages, with a planned move to Firebase Hosting. The app is intended for doctors/clinicians to log procedures, CPD, placements, specialties, locations, hospitals, and related details.

The app must remain polished, safe, and local-first while the data model is still changing. Wider private beta should not start until user accounts and automatic cloud backup/restore are available.

Current confirmed status:
- Current version: V52.
- GitHub main branch is fully up to date and is the latest version.
- Current stack: HTML, CSS, JavaScript, PWA, service worker, manifest, localStorage, Excel export via xlsx.full.min.js.
- Current hosting: GitHub Pages.
- Planned hosting: Firebase Hosting, initially still static/local-first.
- Planned future backend: Firebase Authentication + Firestore/cloud backup later.

2. CORE PRODUCT PRINCIPLES
================================================================================
The user strongly prioritises:
- Clean premium medical aesthetic.
- Multiple small safe batches over large risky updates.
- Future-proof architecture from the beginning.
- Avoiding hacky permanent migration code.
- If migration code is required, prefer temporary migration versions and remove migration code later.
- No patient-identifiable information should be encouraged or required.
- Manual JSON backup must remain available even after cloud backup exists.
- Historical entries must remain readable even if placements, specialties, categories, or procedures are hidden, removed, or deleted from management lists.

Important UX preferences:
- User is very sensitive to visual polish inconsistencies.
- No native browser popups; all popups should be themed to match the app.
- Buttons should use consistent spacing, borders, radius, and shadows.
- Avoid aggressive gradients.
- Emerald green is the primary action colour.
- White/secondary buttons should have subtle internal shadow only.
- Floating nav buttons must feel part of the same design system as normal buttons.

6. PROCEDURE SYSTEM ARCHITECTURE
================================================================================
This is critical.

There are two separate concepts:

A. GLOBAL PROCEDURE LIBRARY
- App-provided procedures:
  - Built into the app.
  - Cannot be deleted from the app by the user.
  - Can be added to or removed from individual specialty/category assignments.
  - Can be shown/hidden within categories.
- User-created procedures:
  - Fully editable/deletable from the app.
  - Can be assigned to one or more specialty/category combinations.
  - If deleted from the custom procedure library, they are removed from every specialty/category where currently used.
  - Historical entries must remain intact using saved snapshots of procedure names/details.

B. SPECIALTY/CATEGORY ASSIGNMENT
- Categories contain selected procedures from the global library.
- Procedures can be added to a category.
- Procedures can be removed from a category.
- Procedures can be shown/hidden within a category.
- Removing a procedure from a category only affects future entries.
- Historical entries must still display normally.

Restore default logic:
- Restore default resets a category back to the app-default procedure list.
- Custom procedures are not deleted from the app.
- User-added procedures are removed from that category.
- Restored default procedures become shown/visible.

7. ADD / REMOVE PROCEDURE FLOW
================================================================================
Desired behaviour:
- Add/Remove procedures pages should support bulk selection.
- Already-added procedures should be listed first alphabetically, then not-added procedures alphabetically.
- Procedures should have right-side capsule buttons such as Add, Remove, Selected, Already added.
- User can tap the capsule to flag/unflag multiple procedures.
- A Make changes / Add selected procedures / Remove selected procedures button applies all selected changes.
- Already added procedures should be visible but disabled/marked as Already added when adding.
- Removing from category must not delete from global library.
- Create new procedure button should appear near the top of the add-procedure flow.
- Search box should appear only if the list/category has more than 10 procedures, unless later UX testing suggests always showing it is better.

================================================================================
8. CUSTOM PROCEDURE PLAN
================================================================================
Custom procedure wizard must support:
- Procedure name.
- Duplicate warning: warn but allow creation.
- If duplicate is found while creating from inside Add procedure to category, offer to add the existing procedure instead.
- If created from Custom Procedures page, do not auto-assign to any category.
- If created from inside a category, ask/allow adding it to that specialty/category by default.

Custom Procedures page:
- Accessible from Specialties & procedures.
- Lists all user-created procedures.
- Shows usage for each custom procedure, e.g. Emergency Medicine → Joint aspiration, Critical Care → Other, or Not currently used.
- Tapping a custom procedure opens view/edit/delete.
- Delete custom procedure removes it from the custom procedure library and all category assignments, but historical entries remain intact using snapshots.
- Custom Procedures page should allow creation of a custom procedure without assigning it by default.
- Custom procedure detail page should have Add to Specialty/Category button allowing the user to choose specialty then category.

Procedure wizard page types, version 1:
- Single-choice button pages.
- Optional free-text notes page.
- Multi-choice may be needed for Complications.

Premade/tickable template pages:
- Site.
- Technique.
- Role.
- Supervision.
- Outcome.
- Number of attempts.
- Complications.
- Optional notes.

Important page-template decisions:
- Date performed, Placement, Specialty, Hospital, Location, and Procedure are always assumed for any procedure and should not appear as tickable template options in the custom procedure wizard.
- Site and Technique should be special named templates, but the user can define the options for each procedure.
- Role, Supervision, Outcome, and similar standard fields can use app-defined options.
- Complications should probably have an extensive/default list plus Other free text, but options may need to vary by procedure.
- Template pages should be filterable/exportable later.
- Fully custom pages should be exportable but not filterable initially.

================================================================================
9. SPECIALTIES / PLACEMENTS / WIZARD FLOW
================================================================================
Placement decisions:
- Users can create placements with a name and date range.
- Placement can be Not linked to placement.
- Placement applies to procedures and likely CPD.
- Placement page in add-entry wizard should show user-defined visible placements.
- Placements button exists on home for add/edit/delete/show/hide.
- Within the wizard, Add placement and Show/hide placements may exist; deletion should remain on the home Placements management page.

Specialty decisions:
- Specialties are app-defined initially: Critical Care, Anaesthetics, Emergency Medicine, Acute Medicine.
- Surgery may come later, not necessarily included now.
- Specialty page should not have add/delete buttons.
- Show/hide specialties should exist.
- Specialties drive specialty-specific procedure/category lists.

Procedure add-entry flow target:
Date → Placement → Specialty → Hospital → Location → Procedure category → Procedure → procedure-specific follow-up pages.

Procedure page should include a search box when useful, especially for long lists.

12. EXPORT / SUMMARIES PLAN
================================================================================
Filtered Excel export:
- Export Excel file should open export options first.
- Filters should include date range, placement, specialty, procedure vs CPD vs all, and later draft/final status if relevant.
- Default should remain export everything.
- Goal is to avoid sending years of irrelevant entries to supervisors.

Filtered summaries:
- Summaries page should allow filtering by date range, placement, specialty, procedure category, and procedure/CPD subgroup.
- Summaries should reuse the same filtering logic as Excel export.

13. FIREBASE / BETA ROADMAP
================================================================================
Beta goal:
Get to a safe private beta as soon as possible, but do not invite wider users until accounts and automatic cloud backup/restore are working.

Revised high-level sequence:
1. Stabilise current V52 situation.
2. Add permanent handover/versioning/testing workflow.
3. Move hosting to Firebase Hosting while still static/local-first.
4. Set up Firebase preview channels and/or GitHub PR preview deployments.
5. Continue procedure-library work.
6. Do data robustness/future-proofing pass before cloud data.
7. Add Firebase Authentication.
8. Add automatic whole-logbook cloud backup.
9. Add restore-from-cloud.
10. Add minimum privacy/security/legal wording.
11. Small private beta with 3–10 colleagues.
12. Fix beta issues.
13. Later: certificate attachments, desktop/review screens, supervisor export/view, donation link, analytics, advanced sync.

Automatic cloud backup target:
- App works locally.
- When an entry is added/edited/deleted, the whole logbook is backed up to the user’s Firebase account.
- Clear statuses: Cloud backup up to date; Cloud backup pending; Last cloud backup date/time; Cloud backup failed — local data is still safe.
- Manual JSON backup remains available.

Restore-from-cloud target:
- User installs app on new device.
- User logs in.
- User restores cloud backup.
- If both local and cloud data exist, offer clear choices first; avoid complex merging initially.

Privacy/security before private beta:
- No patient-identifiable data notice.
- Privacy policy page.
- Explain what is stored locally.
- Explain what is stored in Firebase.
- Explain manual backup/export options.
- Make clear users remain responsible for not entering patient-identifiable data.

15. RECOMMENDED REFACTOR PLAN
================================================================================
Do not split app.js as a giant first step. That is risky.

Recommended staged approach:
1. Add repo hygiene files first:
   - HANDOVER.txt.
   - CHANGELOG.md or RELEASE_NOTES.md.
   - TESTING_CHECKLIST.md.
   - Optional CONTRIBUTING_FOR_CODEX.md / AGENTS.md with coding rules.
2. Add simple automated checks:
   - JavaScript syntax check with node --check app.js.
   - Optional lightweight smoke checks later.
3. Move to Firebase Hosting and preview channels.
4. Only then begin modularisation gradually.

Suggested future module boundaries:
- src/main.js — app bootstrap.
- src/state.js — state management/localStorage/data version.
- src/navigation.js — page transitions/floating nav/warnings.
- src/ui.js — reusable UI rendering helpers/buttons/cards/popups.
- src/placements.js — placement management.
- src/specialties.js — specialties and visibility.
- src/procedures/library.js — global procedure library.
- src/procedures/categories.js — specialty/category assignments.
- src/procedures/custom.js — custom procedures.
- src/entryWizard.js — add/edit entry wizard.
- src/export.js — Excel/JSON export/import.
- src/summary.js — summaries/filtering.
- src/firebase.js — future auth/cloud backup.

Use ES modules if keeping the app build-free, but update index.html and service worker cache carefully. Consider a proper build tool later, but not until the app is stable enough to justify it.
